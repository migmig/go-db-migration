# PRD: 오류 수정 및 테스트 커버리지 +15% 향상 계획 (v27)

## 1) 배경
현재 `go test ./... -coverprofile=coverage.out` 실행 시 `internal/bus` 패키지에서 `go: no such tool "covdata"` 오류가 발생해 전체 커버리지 수집이 비정상 종료된다. 또한 최근 측정된 전체 statement 커버리지는 **56.5%**로, 회귀 방지를 위한 안정적인 품질 기준 수립이 필요하다.

## 2) 목표

### 2.1 기능 목표
1. 커버리지 수집 오류(`covdata`)를 제거하여 `go test ./... -coverprofile=coverage.out`가 안정적으로 완료되도록 한다.
2. 전체 테스트 커버리지를 **+15%p(56.5% → 71.5% 이상)**까지 단계적으로 향상한다.

### 2.2 비기능 목표
- 테스트 실행 시간을 팀 개발 루프를 해치지 않는 범위로 유지한다(기준: CI 기준 2배 이내 증가).
- flaky 테스트(간헐 실패) 비율 0%를 유지한다.

## 3) 문제 정의
- `internal/bus`는 이벤트 버스 핵심 로직임에도 테스트 부재로 인해 커버리지 계측 과정에서 실패 신호가 발생했다.
- 저커버리지 영역(`dbmigrator`, `internal/dialect`, `internal/web/ws`, `internal/web`)에 대한 우선순위 기반 개선 계획이 부재하다.
- 커버리지 게이트가 없어 증가 목표가 지속적으로 유지되지 않는다.

## 4) 범위

### In Scope
- `internal/bus` 테스트 추가 및 커버리지 수집 오류 해소.
- 저커버리지 핵심 패키지 테스트 보강.
- CI/로컬에서 재사용 가능한 커버리지 측정 커맨드 표준화.
- 커버리지 목표(71.5% 이상) 달성을 위한 단계별 일정 수립.

### Out of Scope
- 대규모 아키텍처 리팩터링.
- e2e 성능/부하 테스트 체계 전면 도입.

## 5) 요구사항

### 5.1 오류 수정 요구사항
1. `internal/bus`에 단위 테스트를 추가해 publish/subscribe 기본 동작을 검증한다.
2. `go test ./... -coverprofile=coverage.out` 실행이 오류 없이 종료되어야 한다.
3. `go tool cover -func=coverage.out`로 전체 커버리지 산출이 가능해야 한다.

### 5.2 커버리지 +15%p 요구사항
1. 1차 집중 대상 패키지: `dbmigrator`, `internal/dialect`, `internal/web/ws`, `internal/web`.
2. 각 패키지별 테스트 시나리오(정상/에러/엣지 케이스) 최소 3개 이상 추가.
3. 회귀 방지를 위해 CI에 최소 커버리지 기준을 단계적으로 도입:
   - Phase 1: 60%
   - Phase 2: 66%
   - Phase 3: 71.5%

## 6) 실행 계획 (4주)

### Week 1 — 오류 수정 및 기반 정리
- `internal/bus` 단위 테스트 추가.
- 커버리지 수집 커맨드 표준화(`go test ./... -coverprofile=coverage.out`).
- 베이스라인 재측정 및 리포트 자동 생성 스크립트 초안 작성.

### Week 2 — 핵심 로직 테스트 확장
- `internal/dialect` 파서/분기 로직 테스트 확대.
- `dbmigrator` 엔트리 로직의 인자/오류 경로 테스트 보강.

### Week 3 — 웹 계층 안정화
- `internal/web`, `internal/web/ws`의 핸들러/이벤트 스트리밍 예외 케이스 강화.
- 목(mock) 기반 실패 시나리오(연결 끊김, 잘못된 페이로드) 추가.

### Week 4 — 목표 수렴 및 게이트 적용
- 남은 저커버리지 영역 보완.
- CI 최소 커버리지 71.5% 게이트 적용.
- 최종 결과 리포트 배포.

## 7) 성공 지표 (KPI)
- 전체 statement coverage: **71.5% 이상**.
- `go test ./... -coverprofile=coverage.out` 성공률: 100%.
- CI flaky 테스트: 0건.

## 8) 리스크 및 대응
- **리스크:** 외부 의존 컴포넌트(네트워크/DB)에 따라 테스트가 불안정할 수 있음.
  - **대응:** 단위 테스트는 mock/stub 우선, 통합 테스트는 태그 분리.
- **리스크:** 테스트 증가로 CI 시간 증가.
  - **대응:** 병렬 실행 및 무거운 테스트 분리.

## 9) 산출물
- 오류 수정 코드(`internal/bus` 테스트).
- 커버리지 향상 테스트 코드(PR 묶음).
- 주차별 커버리지 추적 리포트.
- 최종 71.5% 달성 검증 로그.
